The STM32L432KCU6

32-bit ARM Cortex-M4 core running at 80MHz

  • Layout
    Digital IN and OUT: pins D0 to D6, D9 to D13 and A0 to A3 and A6 to A7
    Analogue IN: pins A0 to A6 and D3 can also be configured for analogue input
    Analogue OUT: pins A3 and A4 can be configured for analogue output
    PWM OUT (Pulse Width Modulation): pins D0, D1, D9, D10, A2, A5
    Various other pins can be configured for serial communication such as CAN, SPI, I2C and UART connection.
  • Potential irregularities:
    • Pins A4 and A5 are only ADC input
    • Pins D7 and D8 are not used.
    • The pins marked PWMx/yN are inverted waveforms of the PWMx/y outputs.
    • For PWM, the x represents a timer and y is channel, so if you want a different frequency, you must use a different timer, i.e. PWM1 and PWM16. If you are happy with the same frequency but want different duty cycles, you can use PWM1/1, PWM1/2 etc…

The real beauty of the development board is the simplicity with which the microcontroller can be
programmed via open source compiler environments like Zephyr or FreeRTOS. These come with
extensive application programming interface (API), which is a large set of building blocks (libraries) which are effectively C++ utilities which allows programs to be easily, and quickly developed.

Once compiled, programs are easily flashed to the board via a USB connector, which can also be used to power the board from the 5V USB output. There are 3 LEDs on the board, one for ‘power on’ (red LED2 next to pin D12), a dual colour LED1 for status, and a green LED3 that is connected to pin D13 digital output. LED3 can be driven to test basic functions without the need for any external component connection. However just using a
microcontroller for one LED would be pretty boring, so we will want to attach other stuff…

To connect external components to the microcontroller, you will need to plug the Nucleo board into a breadboard, similar to the one shown in Figure 2 below. Nucleo Programming & Zephyr RTOS

You do not need to write code from scratch in the exam. However, you MUST be able to read code snippets and explain what they do physically (e.g., generating waveforms, reading inputs, triggering interrupts).

1. The 3 Core Files

When programming the Nucleo board in the Zephyr RTOS environment, a project is always split into three specific files. You should know what each does:

  1. prj.conf (Configuration): Turns on the internal MCU subsystems. If you don't enable it here, the compiler throws an error.
  2. .overlay (Device Tree): Maps physical hardware pins to software variables. This is where you configure pull-up resistors, I2C addresses, and PWM polarities.
  3. main.c (The C++ Code): Contains the actual logic, while(1) infinite loops, and API commands.

3. Reading the Overlay File (.overlay)

The overlay file uses "nodes" to define hardware.
Defining a Digital Input (Switch):

button1: button1 {
    /* Maps to pin PA8 (D9). Enables internal pull-down. Active state is HIGH. */
    gpios = <&gpioa 8 (GPIO_PULL_DOWN | GPIO_ACTIVE_HIGH)>; 
};

Defining an I2C Device Address:

&i2c1 {
    status = "okay";
    myTempSensor: myTempSensor@4A {
        compatible = "i2c-device";
        reg = < 0x4A >;  /* <--- This is the unique 7-bit I2C Address! */
    };
};

GPIO & Port Masking:

// Basic output
gpio_pin_set_dt(&led1, 1);  // Sets led1 to HIGH (3.3V)
gpio_pin_set_dt(&led1, 0);  // Sets led1 to LOW (0V)

// Port Masking (Often used for 7-Segment Displays!)
gpio_port_set_masked(led7, 0xFF, 0x3F);
Note
  • What is Port Masking? Instead of setting 8 pins individually, gpio_port_set_masked changes an entire block of pins at once. 0xFF (binary 11111111) is the "mask" that selects the first 8 pins. 0x3F is the actual data being written to them.

Timers & Delays:

k_sleep(K_MSEC(100)); // Puts the MCU to sleep for 100 milliseconds
wait(0.25);           // Older Mbed syntax: Waits for 0.25 seconds (250ms)

DAC/ADC/PWM Operations:

// PWM

// ADC (Analog to Digital)
